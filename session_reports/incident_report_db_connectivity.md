**التاريخ:** 03 فبراير 2026
**المشروع:** Social Commerce Platform
**الحالة:** ✅ مكتمل (Resolved)

---

## 1. وصف المشكلة (Problem Description)
فشل التطبيق المحلي (Local Environment) في الاتصال بقاعدة البيانات المستضافة سحابياً (Neon PostgreSQL) أثناء عمليات التطوير والـ Deployment (`prisma db push`).

**رسالة الخطأ:**
```
Error: P1001: Can't reach database server at `...neon.tech:5432`
Ping statistics: Packets: Sent = 4, Received = 0, Lost = 4 (100% loss)
```

---

## 2. التحليل الهندسي (Engineering Analysis)

### أ. التشخيص (Diagnosis)
تم إجراء اختبارات العزل التالية:
1. **التحقق من بيانات الاعتماد:** تم التأكد من صحة `DATABASE_URL` (اسم المستخدم، كلمة المرور، المضيف).
2. **اختبار خادم النطاق (DNS):** تم حل اسم النطاق بنجاح إلى IP `23.21.74.185`.
3. **اختبار الاتصال المباشر (ICMP Ping):** فشل بنسبة خسارة 100%.
4. **اختبار المنفذ (TCP Port Check):** فشل الاتصال عبر المنفذ `5432` باستخدام `Test-NetConnection`.
5. **تغيير مزود الخدمة:** تم اختبار شبكتين مختلفتين (WiFi منزلي + بيانات هاتف)، وكلاهما أظهر نفس الحظر.

### ب. السبب الجذري (Root Cause)
**حظر المنفذ 5432 (Port Blocking):**
مزود خدمة الإنترنت (ISP) أو جدار الحماية (Firewall) في بيئة المستخدم يقوم بحظر حركة المرور الصادرة (Outbound Traffic) عبر المنفذ القياسي لقواعد بيانات PostgreSQL (`5432`). هذا إجراء أمني معتاد في بعض الشبكات العامة أو الشركات، وأحياناً لدى مزودي الخدمة في مناطق معينة.

## 3. الحلول والتوصيات (Solutions & Recommendations)

### الحل أ: استراتيجية "HTTP Tunneling" (موصى به فنياً) 🛠️
استخدام تقنية للاتصال بقاعدة البيانات عبر منفذ الويب القياسي (`443` HTTPS) أو المنافذ غير المحجوبة، بدلاً من اتصال TCP المباشر.

**التنفيذ المقترح:**
- **الانتقال إلى Supabase:** يوفر Supabase اتصالاً عبر المنفذ `6543` (Transaction Mode) أو `5432`، ولكنه يدعم أيضاً الاتصال عبر `443` في بعض التكوينات، والأهم أنه أقل عرضة للحجب في المنطقة الحالية.
- **Neon Serverless Driver:** استخدام مكتبة `@neondatabase/serverless` التي تتصل عبر بروتوكول WebSocket (يعمل عادةً عبر `443`).

### الحل ب: التحايل على الشبكة (Workaround) 🔄
- **VPN:** استخدام شبكة افتراضية خاصة لتشفير الاتصال وتجاوز حجب المنفذ (فشل في الاختبار الأولي، قد يتطلب بروتوكول مثل WireGuard عبر UDP 443).
- **نقطة اتصال مختلفة:** البحث عن شبكة (Hotspot) لا تفرض سياسات حجب صارمة (تمت التجربة وفشلت، مما يشير لحجب واسع النطاق من المزودين).

### الحل ج: التطوير المحلي (Offline Development) 🏠
- استخدام `Docker` لتشغيل نسخة PostgreSQL محلية أثناء التطوير، ومزامنتها مع السحابة فقط عند النشر (Deployment).

## 5. الحل المعتمد والنتائج (Resolution & Results) ✅

**الحل النهائي:** استخدام **Railway PostgreSQL** مع تكوين أمني محدد.

**لماذا نجح؟**
1. **تجاوز الحجب (Port Evasion):**
   - Railway يوفر "Public Proxy URL" يستخدم منفذ عشوائي (في حالتنا `48259`).
   - هذا المنفذ غير موجود في "القائمة السوداء" الافتراضية لمزود الخدمة (بخلاف `5432`).
2. **تشفير SSL (SSL Encryption):**
   - الاتصال الأولي فشل (`P1001`) رغم أن المنفذ مفتوح.
   - تبين أن Railway يتطلب اتصالاً مشفراً إجبارياً، بينما إعدادات Prisma الافتراضية تحاول الاتصال بملف شهادة غير موجود.
   - **الحل:** إضافة `?sslmode=require` إلى رابط الاتصال.

**النتيجة النهائية:**
- ✅ نجاح اختبار الاتصال عبر `test-db.js`.
- ✅ نجاح `prisma db push` ومزامنة المخطط.
- ✅ الاتصال مستقر من الشبكة المحظورة سابقاً.

```bash
# Connection String Format Used
DATABASE_URL="postgresql://user:pass@...railway.net:48259/railway?sslmode=require"
```

**مصير Neon (Neon Status):**
- ❌ **تم الاستبعاد حالياً:** لأنه يعتمد حصرياً على المنفذ `5432` للاتصالات القياسية (TCP)، وهو المنفذ المحجوب.
- 💡 **الحل البديل المعقد:** يمكن استخدامه فقط عبر `Serverless Driver` (WebSocket)، لكنه يتطلب تغييرات في كود `Prisma`. تم تفضيل Railway لأنه يعمل دون تغيير الكود.
